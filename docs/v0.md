# Yuzu Discord Bot Requirements v0

## 1. 目標

建立一個可擴充、可長期維護的 Discord Bot：**Yuzu**。

Yuzu 需支援：

- Slash commands
- 關鍵字統計與排行榜
- 可擴充成就系統
- Per-guild 功能設定
- Auto role
- Self color role
- Docker Compose 部署
- PostgreSQL 本地持久化儲存
- GitHub Actions CI
- 自動 changelog / release 流程
- 自動資料庫備份

## 2. 設定管理

### `.env`

`.env` 僅存放部署環境相關的敏感資訊，例如：

```env
DISCORD_TOKEN=
DATABASE_URL=
```

可依實際需求新增其他 secrets。不危險的機器人設定不得放入 `.env`，應由 config 管理。

## 3. 部署

使用 Docker Compose

## 4. 資料庫備份

需支援 PostgreSQL 自動備份：

- 每日備份一次
- 備份檔以時間命名
- 備份檔儲存在宿主機目錄，不存在 docker volume 內
- 僅保留最近 7 天
- 自動刪除過期備份

## 5. CI / Release

### CI

每次 push 或 PR 必須執行：

- Format check
- Lint check
- Build check

### Release

需提供簡單、可重現的版本發布流程，並支援：

- 標準化 version bump
- 產生 changelog
- GitHub release

## 6. Slash Command 系統

Command system 需支援：

- 多個指令
- 指令分類
- 權限檢查
- Cooldown
- 統一錯誤處理，錯誤不得導致 bot crash
- 易於新增新指令

所有 command 執行結果需記錄 log。

## 7. 指令分類

### 7.1 Info

需提供：

- Bot 狀態，例如 ping
- Bot 資訊
- Guild 資訊
- User 資訊
- 本 guild 所有設定狀態
- GitHub changelog 連結
- 指令分類、列表與說明

### 7.2 Achievements

需提供：

- 列出所有成就
- 查詢使用者已獲得成就與日期
- 成就排行榜

### 7.3 Settings

管理員可查詢與修改 guild 設定。
所有 settings 修改都必須記錄 log。

### 7.4 Leaderboard

需提供統計排行榜。

### 7.5 Owner Only

僅 bot owner 可使用：

- 測試用指令
- 健康檢查
- 讓 bot 在目前頻道發送指定訊息
- 手動修改指定使用者的統計值

## 9. 關鍵字系統

### 9.1 關鍵字統計

當訊息命中（包含或其他方式判斷）指定關鍵字時，累積使用者對應統計值。

需求：

- 支援多種統計類型
- 支援多個觸發關鍵字
- 支援隨機回覆集合
- 可新增新統計類型
- 受 guild 設定控制
- 不應統計 bot 自己或其他 bot 的訊息
- 錯誤不得使 bot crash

範例：

使用者說：

```txt
破防
破大防
```

Yuzu 會：

- 增加該使用者的 `breakdown` 統計值
- 依設定隨機回覆一句預設內容
- 觸發相關成就判定

### 9.2 關鍵字回覆

當訊息命中指定關鍵字時，回覆預設內容。

需求：

- 支援多種回覆類型
- 支援多個觸發關鍵字
- 支援隨機回覆集合
- 可新增新回覆類型
- 受 guild 設定控制
- 可設定 cooldown，避免洗頻
- 不應回覆 bot 自己或其他 bot 的訊息

## 10. 成就系統

成就系統需支援：

- 不限於計數門檻型成就
- 可由事件觸發判定
- 同一使用者同一成就僅發放一次
- 可新增新成就規則而不修改核心邏輯
- 可查詢所有成就
- 可查詢使用者已獲得成就與取得日期
- 可建立成就排行榜

### 成就範例

```txt
breakdown_count_100
```

條件：

- 使用者的 `breakdown` 統計達 100 次

```txt
keyword_in_channel
```

條件：

- 使用者在指定頻道說過指定關鍵字

## 14. Logging

Yuzu 需支援兩種 log：

- Runtime logs：用於除錯與監控 bot 運行狀態
- Audit logs：用於追蹤重要管理操作

### 14.1 Runtime Logs

Runtime logs 需輸出至 stdout / stderr，並支援 log level。

不得輸出敏感資訊。

需記錄：

- 啟動資訊
- PostgreSQL 連線狀態
- 指令執行結果
- Auto role assign 結果
- Color role 建立、重用、分配與清除
- 重大錯誤

Log level 至少支援：

- `debug`
- `info`
- `warn`
- `error`

Runtime logs 僅作為運行時除錯用途，可能因 log rotation 或 container 重建而消失，不應視為永久紀錄。

### 14.2 Audit Logs

重要管理操作需記錄為 audit logs，並持久化保存。

需記錄：

- Guild 設定變更
- Owner 手動統計值修改
- Auto role 設定變更
- Color role 設定變更
- 其他會影響 guild 或使用者資料的重要操作

Audit logs 至少需包含：

- 操作人
- 目標 guild
- 目標使用者或資源，若有
- 操作類型
- 操作內容
- 操作時間

所有 audit logs 不得包含敏感資訊。

## 15. 架構要求

架構需滿足：

- 容易新增 slash command
- 容易新增統計類型
- 容易新增關鍵字回覆
- 容易新增成就規則
- 容易新增 guild setting
- 功能解耦
- 避免集中式巨大條件判斷
- 核心邏輯與 Discord adapter 盡量分離
- Discord event handler 不應直接塞大量 business logic
